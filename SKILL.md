# 主题生成 AI 录播课 Skill

## 核心约定

默认用户只提供一个课程主题。

除非主题真的不清楚，不要反复追问课程类型、时长、受众、交付物。默认按下面的标准执行：

- 课程类型：录播型 AI 实操课。
- 受众：King 的 AI 课学员。
- 教学风格：先讲“为什么”，再讲“怎么做”。
- 本地保存：统一放到 King 知识库的 `AI课项目\AI课` 下面。
- 飞书交付：创建或更新飞书学员讲义，并挂到对应的飞书学员资料栏目下。
- 课件理解：用户说“做课件”“生成课件内容”“形成上下文”，默认理解为课程内容生产，不是做 PPT、HTML 或视觉课件。只有用户明确要 HTML、PPT、幻灯片、视觉设计时，才生成这些文件。

执行前必须先读取：

```text
references/config.md
```

这个文件里写本地路径和飞书目录规则。

## 默认交付物

每次做课，至少产出三类东西。

### 1. 讲师手卡

讲师手卡用于老师录课时直接照着讲。

保存位置：

```text
D:\king知识库\king知识库\AI课项目\AI课\<课程名>\<小节名>\讲师手卡.md
```

要求：

- 用中文写。
- 必须是能直接口播的讲稿，不要写成大纲。
- 不写“这里要讲”“可以这样说”“提醒学生”这种老师提示语。
- 要包含场景、痛点、为什么要学、实操流程、提示词、预期结果、常见失败和补救方法。

### 2. 学员讲义本地源文件

学员讲义本地源文件也放在同一个小节目录里。

保存位置：

```text
D:\king知识库\king知识库\AI课项目\AI课\<课程名>\<小节名>\
├── 学员讲义.md
└── 学员讲义.xml
```

不要把正式源文件放到：

```text
D:\king项目\content
```

`D:\king项目\content` 只能作为临时输出或旧项目兼容位置，不作为正式归档入口。

### 3. 飞书学员讲义

要求：

- 创建或更新飞书文档。
- 如果用户明确说已经在飞书里手动修改过文档，后续默认不要再覆盖更新该飞书文档。先把用户的飞书手改视为最新版本和事实来源；除非用户明确要求“覆盖飞书”“同步到飞书”或“用本地文件重写飞书”，否则只更新本地源文件、GitHub 或给出修改建议。
- 覆盖更新飞书前，先确认不会覆盖用户手动改过的内容。若无法确认，必须先提醒用户存在覆盖风险，并改用读取飞书内容、局部补丁或只给修改建议的方式。
- 放到对应的飞书学员资料栏目下面。
- 如果是“手把手实操 AI 项目”，就挂到飞书“手把手实操 AI 项目”对应项目子页面下。
- 讲义顶部和结尾都要有返回目录链接。
- 如果课程给学员提供图片、素材包、示例文件、提示词文件、脚本、模板等教学材料，必须把这些材料同步上传到 GitHub 课程资料仓库，并在飞书讲义或课程说明里写清下载位置。
- 课程涉及的教学图片或示例图片，必须同时插入到对应飞书讲义正文里，不能只写本地路径或只放 GitHub 链接。
- 最终回复用户时，必须给飞书链接。

### 4. GitHub 学员素材仓库

凡是提供给学员使用的文件，都要同步上传到 GitHub。默认规则：

- 用一个统一的课程资料仓库保存学员可下载材料，按课程名建立子目录。
- 每门课至少包含：课程说明、每节学员讲义、素材包、必要的 README 或索引说明。
- 图片、示例文件、模板、脚本和提示词文件都放进对应课程目录，不要散落在仓库根目录。
- 默认不要上传只给老师录课用的讲师手卡，除非用户明确说要给学员看。
- 如果仓库不存在，优先创建私有仓库，避免课程资产被误公开；用户明确要求公开时再改为 public。
- GitHub 上传完成后，回填仓库链接到本地课程说明和飞书父页面或对应讲义。

## 做课流程

### 1. 先把主题变成课程定位

拿到主题后，先在内部判断四件事：

- 这节课一句话讲什么。
- 学员遇到的真实问题是什么。
- 学完以后能做什么。
- 这节课不讲什么。

不要把这部分内部分析全部发给用户，除非用户要看。

### 2. 先写场景和痛点，不要先讲概念

每节课开头必须从具体场景讲起。

要讲清楚：

- 学员现在遇到什么情况。
- 以前手动怎么做。
- 为什么麻烦。
- 为什么这件事适合用 AI 做。

示例口吻：

```text
很多人不是不会用 AI，而是每次换一个任务，都要重新想提示词、重新试工具、重新判断结果对不对。真正麻烦的不是某一步不会，而是整个流程不能复用。
```

如果用户给了具体案例、行业、产品、数据集，只能把它放在演示或举例部分。除非用户明确说这个案例就是课程主题，否则不要把一个例子抬成整节课的定位。

### 3. 原理要放在实操前后讲，不要一开头堆概念

不要只写“第一步、第二步、第三步”。

每个关键方法都要讲清楚：

```text
方法是什么 -> 为什么要这样 -> 不这样会怎样 -> 换个场景怎么迁移
```

实操课里尤其要解释：

- 为什么不能只问“帮我全面分析”。
- 为什么一个提示词最好只解决一个任务。
- 为什么要先检查输入材料，再让 AI 下结论。
- 为什么 AI 的结论要能找到依据。
- 为什么最终结果要转成汇报、教学或行动语言。

### 4. 设计实操流程

每一步都要包含：

- 这一步叫什么。
- 解决什么问题。
- 为什么现在做。
- 老师怎么讲。
- 学员复制哪段提示词。
- 正常会得到什么结果。
- 如果 AI 跑偏，怎么拉回来。

提示词规则：

```text
一个提示词 = 一个任务 + 一个输出格式 + 一个判断标准
```

用户在对话里纠正、确认或新增的课程口径，必须当成课件修改任务处理，不能只在聊天里回答。比如用户确认“某个提示词一段就够，不要拆成两段”“某个模型/分辨率/素材数量要改”“某个章节说法不对”，就要定位受影响的学员讲义、讲师手卡、飞书文档和 GitHub 学员素材说明，按需要同步更新。若飞书文档已被用户手动改过，必须先读取飞书当前内容，以飞书为准做局部更新，保留用户手改内容，不要用本地源文件全文覆盖。

### 5. 写讲师手卡

讲师手卡必须是口播稿，不是提醒稿。

必须包含：

- 课程标题和目标。
- 开场讲法。
- 场景和痛点。
- 为什么要学这节课。
- 核心原理。
- 完整演示流程。
- 每一步提示词。
- 每一步为什么这么设计。
- 常见问题和补救话术。
- 结尾和作业。

禁止出现：

- “这里要讲……”
- “可以这样说……”
- “提醒学生……”
- “课堂上可以……”
- “这个地方要……”

要直接写成老师能说出口的话。

错误写法：

```text
这里要提醒学生先不要急着生成图片。
```

正确写法：

```text
同学们，这一步我们先不急着生成图片。因为如果参考图还没拆清楚，后面生图模型就只能靠猜。
```

### 6. 写学员讲义

学员讲义要比讲师手卡更干净。

必须包含：

- 这节课教什么。
- 为什么要学。
- 适用场景。
- 核心原则。
- 操作步骤。
- 可复制提示词。
- 注意事项。
- 常见错误。
- 作业。
- 怎么迁移到自己的场景。

不要写老师专用的节奏提示。

### 7. 对齐检查

发布前必须检查：

- 学员讲义每个章节，讲师手卡里都有对应讲法。
- 学员讲义里的表格、清单、提示词、作业，讲师手卡里都讲到了。
- 讲师手卡里新增的重要概念，如果学员需要记住，也要补进学员讲义。
- 两边顺序要一致，不能讲师手卡一套逻辑，学员讲义另一套逻辑。
- 即使只是补充一个课堂改动、素材路径、提示词或局部段落，也必须重新通读学员讲义全文，顺一遍上下文。
- 通读时重点检查：是否出现两套目录、两套路径、两套命名规则、两套实操流程；是否有旧模板残留；是否前半段讲真实案例、后半段又切回通用示例。
- 如果新增内容会改变课堂流程，不能只把新内容插到前面或后面，必须把整篇讲义重排成一条连续路径。

如果学员讲义和讲师手卡对不上，先改本地源文件，再更新飞书。

### 8. 模型选择类课程的特殊规则

如果课程涉及“怎么选模型”，不要让新手默认去评测一堆模型。

正确做法：

- 先给推荐路线。
- 再解释模型能力差异。
- 明确区分“看懂图片”的视觉理解模型和“生成图片”的生图模型。
- 只在必要时讲测试和评估。

讲模型名字时要具体，不要把平台、产品族、模型族混成一个名字。

例如：

- “Gemini”是产品或模型家族，不是单个模型。
- “豆包”是产品或模型家族，不是单个模型。
- “千帆”是平台，不是单个模型。
- “混元”是模型家族或云服务，不是单个模型。

如果模型名称变化快，在学员讲义里加一句边界说明：

```text
模型名称和可用版本会更新，正式使用前以官方模型列表为准。
```

### 9. 保存和发布

执行顺序：

1. 先保存本地源文件到 King 知识库的正确课程目录。
2. 再创建或更新飞书学员讲义。
3. 再把飞书链接挂到对应的飞书学员资料目录。
4. 再把面向学员的教学素材和示例文件上传到 GitHub 课程资料仓库。
5. 再把课程相关图片插入飞书讲义正文，并在飞书里写清 GitHub 下载位置。
6. 最后更新本地索引或链接说明。

普通 AI 课默认挂到 `AI课总目录`。

手把手实操项目课，优先挂到：

```text
飞书：手把手实操 AI 项目
对应项目子页面
课程内容
```

不要只创建飞书散文档。

### 10. 最终回复用户

只汇报最重要的位置：

- 本地课程目录。
- 讲师手卡路径。
- 学员讲义本地路径。
- 飞书学员讲义链接。
- 是否已经挂到飞书学员资料目录。
- 还有哪些配置需要用户确认。

## 质量标准

只有同时满足这些条件，才算完成：

- 讲师手卡能直接照着录。
- 讲师手卡没有老师提示语。
- 学员讲义离开老师也能照着做。
- 学员讲义和讲师手卡完全对齐。
- 学员讲义全文顺读是连贯的，没有旧模板残留、重复说明、上下文断裂、前后路径不一致或两套流程并存。
- 先讲场景和痛点，再讲操作。
- 每个关键操作都解释“为什么”。
- 提示词可以直接复制。
- 飞书讲义已经挂到对应目录。
- 本地源文件已经保存到 King 知识库 `AI课项目\AI课` 下对应课程目录。
- 面向学员的素材包、图片、模板、示例文件已经上传到 GitHub 课程资料仓库。
- 飞书讲义中已经插入相关教学图片，学员不用离开飞书也能看到素材效果。

## 禁止事项

- 不要默认问用户课程时长。
- 不要让用户重复说明交付物。
- 不要只产出大纲。
- 不要把“课件内容”误解成必须做 PPT 或 HTML。
- 不要默认上传讲师手卡到飞书。
- 不要默认把讲师手卡上传到 GitHub 学员素材仓库。
- 不要把老师提示语写进学员讲义。
- 不要把正式源文件放到 `D:\king项目\content`。
- 不要创建一堆飞书散文档却不挂目录。
- 不要只在飞书写本地素材路径，却不上传 GitHub；也不要只上传 GitHub，却不把关键教学图片贴进飞书正文。
- 用户已经在飞书里手动改过的文档，不要再用本地源文件直接 overwrite 覆盖，除非用户明确授权。
